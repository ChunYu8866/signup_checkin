# 華南永昌證券 2026 下半年投資展望會報到系統設計規格

日期：2026-07-16  
狀態：已完成需求確認，待使用者複核書面規格

> **架構更新註記（2026-07-17）**
>
> 前端與後端的通訊方式已由「隱藏 iframe + `postMessage` 橋接」改為「前端直接以 `fetch` POST 呼叫 Apps Script `doPost`（`Content-Type: text/plain` 以避開 CORS preflight）」，目的是改善行動網路下的載入速度與穩定度。此變更的重要安全影響：
>
> - Apps Script 的 `doPost(e)` **無法讀取 HTTP `Origin` 標頭**，因此本文 §3.2、§13.2、§15 所述的「來源白名單（ALLOWED_ORIGINS）」在 `fetch` 路徑下**不再具強制力**，`FORBIDDEN_ORIGIN` 不會由後端主動回傳。部署存取權為「任何人（含匿名）」，等同於端點對外公開。
> - 現存的補償控制仍然有效：查詢僅回傳遮罩姓名、確認報到必須使用查詢後產生的短效憑證、現場報名受 `WALK_IN_ENABLED` 與個資核准雙重閘門控管、寫入使用 `LockService` 並有人數上限。殘餘風險主要是「以已知手機/E-mail 探測是否在名單中」與「對已知聯絡人惡意報到」，對本次單日活動屬可接受範圍。
> - **補償控制新增（2026-07-17）**：**近似速率限制**——同一識別每 10 分鐘至多 12 次查詢、全站每分鐘至多 240 次查詢、現場報名全站每分鐘至多 20 筆，超量回 `BUSY` 交由前端等候室退避，抑制名單探測與現場報名灌水（`CacheService` 計數非原子，屬濫用抑制而非精確配額）。曾評估「報到時間窗」（活動期間外回絕請求）方案，因營運彈性考量由主辦方決定不採用；活動結束後應直接封存或停用 Web App 部署以關閉端點。另補「**維運函式守門**」：`initializeSheet`／`validateDeployment`／`warmIndexes`／`refreshIndexes` 以 `Session.getActiveUser()` 驗證執行者，匿名 Web App 情境（含經 Bridge 頁面的 `google.script.run`）一律拋出 `OPERATOR_ONLY`，防止匿名者讀取名單筆數或濫用索引重建耗損配額。
> - `doGet` / `Bridge.html` / `bridge-client.js` 暫時保留作為備援；若 `fetch` 路徑於活動前實測穩定，可再行移除。若未來需要真正的來源限制或身分驗證，需改用具伺服器端標頭存取能力的後端（如 Cloud Run）。

## 1. 目標與範圍

本系統提供「2026 下半年投資展望會」行動報到服務。參加者掃描共用 QR Code 後，透過 GitHub Pages 開啟 RWD 網頁，再由 Google Apps Script 查詢或寫入 Google Sheet。

系統涵蓋兩條流程：

- 預先報名者使用手機後 8 碼或 E-mail 查詢名單，確認遮罩姓名後報到。
- 未預先報名者填寫姓名、手機與 E-mail，完成現場報名及報到。

本次不製作工作人員後台、報表儀表板、簡訊驗證、寄信通知、QR Code 個人化或嚴格 FIFO 排隊名次。

## 3. 系統架構

### 3.1 GitHub Pages 前端

GitHub Pages 負責所有可見介面、RWD、輸入驗證、流程狀態與虛擬等候室。前端不直接存取 Google Sheet，也不把姓名、手機或 E-mail 寫入 `localStorage`、`sessionStorage`、Cookie 或網址參數。

### 3.2 Apps Script Bridge

Apps Script Web App 輸出一個只負責通訊的 HTML 頁面。GitHub Pages 以隱藏 iframe 載入 Bridge，並透過 `window.postMessage` 傳送請求。Bridge 使用 `google.script.run` 呼叫 Apps Script 伺服器函式。

Bridge 必須：

- 設定 `XFrameOptionsMode.ALLOWALL`，讓 GitHub Pages 可以載入 iframe。
- 驗證每一則訊息的 `event.origin`，只接受部署設定中的 GitHub Pages 正式來源與本機測試來源。
- 驗證訊息格式、操作名稱、請求 ID 與欄位長度。
- 僅回傳給原始 `event.source` 及已驗證的 `event.origin`。
- 拒絕未知來源、未知操作、過期請求及格式不符的資料。

GitHub Pages 的正式來源在部署時由儲存庫名稱決定。正式上線前，部署檢查必須確認 Apps Script 允許清單與實際 GitHub Pages `origin` 完全一致。

### 3.3 Apps Script 後端

Apps Script 負責：

- 正規化手機與 E-mail。
- 查詢既有名單。
- 產生遮罩姓名與短效確認憑證。
- 檢查重複報到。
- 建立現場報名資料。
- 寫入第一次報到時間。
- 建立及失效快取索引。
- 回傳可供前端處理的固定錯誤代碼。

### 3.4 Google Sheet

使用既有試算表：

- 名稱：下半年投資展望會_簽到
- Spreadsheet ID：`179uW_qocdZQ8H-yZNYz3_IhNEyviKWCkBDnrnHZQkQU`
- 工作表：`簽到表`

正式欄位固定為：

| 欄 | 名稱 | 格式與用途 |
| --- | --- | --- |
| A | 姓名 | 預先名單或現場輸入的完整姓名 |
| B | 手機 | `09` 加 8 位數字，共 10 碼 |
| C | E-mail | 去除首尾空白後保存；查詢不分大小寫 |
| D | 報名類型 | `預先報名` 或 `現場報名` |
| E | 報到狀態 | 空白或 `已報到` |
| F | 報到時間 | 台北時區的日期時間值 |
| G | 資料建立時間 | 現場報名建立時間；預先名單可保留空白 |

匯入預先名單時，D 欄統一填入 `預先報名`。F、G 欄使用試算表日期時間值，不保存已格式化的文字。

## 4. 輸入正規化與驗證

### 4.1 手機

- 畫面固定顯示 `09`，參加者輸入後 8 碼。
- 輸入元件使用 `type="text"`、`inputmode="numeric"`、`autocomplete="tel-national"` 與 `maxlength="8"`。
- 前端移除非數字字元，只接受 8 位數字。
- 後端再次驗證，補成 `09XXXXXXXX`。
- Google Sheet 既有資料在查詢時移除空白、連字號及括號，再比對 10 位數字。

### 4.2 E-mail

- 使用 `type="email"`、`inputmode="email"` 與 `autocomplete="email"`。
- 前後空白會被移除。
- 比對時轉成小寫；Sheet 可保留使用者輸入的原始大小寫。
- 前後端都限制長度為 254 字元，並檢查基本 E-mail 結構。

### 4.3 姓名

- 去除前後空白並合併連續空白。
- 必填，長度 2 至 50 個字元。
- 允許中英文、空白、中點與常見姓名連字號；拒絕控制字元及 HTML。

## 5. 使用者流程

### 5.1 首頁

首頁顯示主辦單位、活動名稱、日期、時間、地點及地址，並提供兩個主要入口：

- 我有事先報名
- 我要現場報名

### 5.2 預先報名者

1. 參加者選擇「我有事先報名」。
2. 頁面要求輸入手機後 8 碼。
3. 後端以完整手機查詢。
4. 查到唯一資料時：
   - 若已報到，顯示「您已完成報到」及第一次報到時間。
   - 若尚未報到，顯示遮罩姓名及「確認報到」。
5. 手機查無資料時，頁面改為 E-mail 查詢。
6. E-mail 查到唯一資料時，套用相同的已報到或確認流程。
7. 手機與 E-mail 都查不到時，轉入現場報名，並保留已輸入的手機及 E-mail。
8. 使用者按下「確認報到」後，後端再次核對資料與狀態，再寫入第一次報到時間。

姓名遮罩規則：

- 1 個字：顯示 `○`。
- 2 個字：顯示首字加 `○`，例如 `林○`。
- 3 個字以上：保留第一個與最後一個字，中間每個字改成 `○`，例如 `林○宇`、`歐○○明`。

查詢成功時，前端只收到遮罩姓名、報到狀態、已格式化的報到時間及短效確認憑證，不收到完整姓名、完整手機、完整 E-mail 或試算表列號。

### 5.3 現場報名

1. 參加者填寫姓名、手機後 8 碼及 E-mail。
2. 頁面展開個資蒐集告知，參加者勾選同意後才能送出。
3. 後端先以手機及 E-mail 查詢既有資料。
4. 若任一欄位找到唯一資料，後端不新增列，改回既有參加者的遮罩確認流程。
5. 若手機與 E-mail 分別對應不同資料，或任一欄位對應多筆資料，系統顯示「資料需要由現場工作人員確認」，不自動合併或報到。
6. 兩者都查無資料時，後端新增一列，D 欄填入 `現場報名`，E 欄填入 `已報到`，F、G 欄寫入同一個建立時間。
7. 頁面顯示「現場登記與報到已完成」。

### 5.4 重複報到

同一筆資料再次操作時，後端不新增紀錄，也不覆寫 F 欄。頁面顯示：

- 您已完成報到
- 第一次報到時間，格式為 `yyyy/MM/dd HH:mm`

## 6. 確認憑證與資料保護

查詢成功後，後端產生不可預測的 UUID 確認憑證，並在 Script Cache 保存憑證與資料列參照，效期 5 分鐘。確認報到只能使用該憑證，不能使用前端提供的手機、E-mail 或列號指定資料列。

若憑證過期或遭快取提前清除，前端顯示「確認時間已超過，請重新查詢」，並保留使用者輸入內容。後端不接受重放已使用的憑證。

伺服器與瀏覽器記錄不得輸出完整姓名、手機、E-mail、確認憑證或完整請求內容。錯誤記錄只保存請求 ID、操作類型、錯誤代碼及時間。

## 7. 快取、鎖定與虛擬等候室

### 7.1 查詢索引

資料量上限以 1,000 人設計。Apps Script 使用 Script Cache 保存分片索引：

- 手機與 E-mail 先正規化，再以 SHA-256 雜湊作為索引鍵。
- 索引值只保存對應列號，不保存可直接閱讀的聯絡資料。
- 索引依雜湊前綴分片，每個快取值保持在 Apps Script 單一項目大小限制內。
- 快取未命中或過期時，後端一次讀取 A:G 的實際資料範圍並重建索引。
- 新增現場報名後，後端使相關索引分片失效；下一次查詢會重建。
- 快取只加速查詢，Google Sheet 仍是唯一資料來源。

### 7.2 寫入鎖定

查詢不取得全域鎖。確認報到及現場報名使用 `LockService.getScriptLock().tryLock()`：

- 取得鎖後，後端重新讀取目標資料列或重新比對手機與 E-mail。
- 後端只寫入尚未完成的欄位。
- 寫入後呼叫 `SpreadsheetApp.flush()`，再釋放鎖。
- 未在短時間內取得鎖時，回傳 `BUSY`，不在伺服器內長時間等待。

### 7.3 虛擬等候室

前端收到 `BUSY`、可重試的執行失敗或網路暫時錯誤後，切換為虛擬等候室：

- 顯示「目前報到人數較多，系統正在為您安排報到，請勿關閉頁面」。
- 保留本次請求與輸入內容，停用重複送出。
- 使用 2、4、8、12、16 秒的遞增間隔，每次加入隨機延遲，避免裝置同時重送。
- 自動重試總時間不超過 60 秒。
- 60 秒後仍失敗，顯示「再次嘗試」與「返回修改資料」。
- 不顯示虛構的排隊人數、名次或等待分鐘數。
- 每次重試沿用同一個請求 ID；後端以資料狀態實作冪等性。

虛擬等候室只能緩衝瞬間流量，不能保證嚴格先到先服務。若主辦單位需要可驗證的 FIFO 名次，系統必須改用 Cloud Run、Firestore 或專用佇列服務。

## 8. API 訊息契約

每個 Bridge 訊息包含：

- `version`：固定為 `1`
- `requestId`：前端產生的 UUID
- `action`：允許的操作名稱
- `payload`：該操作需要的欄位

允許的操作：

- `healthCheck`
- `lookupByPhone`
- `lookupByEmail`
- `confirmCheckIn`
- `registerWalkIn`

統一回應包含：

- `version`
- `requestId`
- `ok`
- `code`
- `data`

主要回應代碼：

| 代碼 | 前端行為 |
| --- | --- |
| `FOUND` | 顯示遮罩姓名與確認按鈕 |
| `NOT_FOUND` | 手機改查 E-mail；E-mail 改現場報名 |
| `ALREADY_CHECKED_IN` | 顯示第一次報到時間 |
| `CHECKED_IN` | 顯示報到成功 |
| `WALK_IN_REGISTERED` | 顯示現場登記與報到成功 |
| `DATA_CONFLICT` | 請洽現場工作人員 |
| `TOKEN_EXPIRED` | 保留輸入並重新查詢 |
| `BUSY` | 進入虛擬等候室 |
| `INVALID_INPUT` | 聚焦第一個錯誤欄位 |
| `FORBIDDEN_ORIGIN` | 停止操作並顯示系統設定錯誤 |
| `SYSTEM_ERROR` | 顯示可重試訊息與請求 ID |

- 採手機優先設計，支援 320px 以上寬度。
- 手機版單欄滿寬，保留至少 16px 邊距。

前端明確呈現：首頁、手機查詢、E-mail 查詢、遮罩確認、現場報名、查詢中、虛擬等候室、成功、已報到、資料衝突、網路中斷及系統錯誤。

成功或錯誤不能只依賴顏色。每個狀態同時使用文字、圖示及可聚焦的狀態標題。

## 10. 無障礙與操作細節

- 所有欄位都有可見標籤、提示文字與錯誤訊息。
- 表單錯誤發生時，焦點移到第一個錯誤欄位。
- 動態狀態使用 `aria-live`，避免朗讀每秒倒數造成干擾。
- 所有操作可使用鍵盤完成，焦點順序與視覺順序一致。
- 支援 `prefers-reduced-motion`，降低載入及成功動畫。
- 送出期間停用主要按鈕並顯示處理狀態。
- 瀏覽器返回鍵可回到前一步，但成功頁不重新送出請求。

## 11. 個資蒐集告知

現場報名頁在送出前顯示可展開的個資蒐集告知，至少包含：

- 蒐集機關：華南永昌綜合證券股份有限公司。
- 蒐集目的：本活動報名、身分確認、報到管理及必要聯繫。
- 個資類別：姓名、手機號碼及 E-mail。
- 利用期間、地區、對象及方式：由主辦單位依其核准的活動個資政策填入正式內容。
- 當事人可行使的權利及聯繫方式。
- 不提供資料的影響：無法完成現場報名及報到。

送出按鈕上方提供未預先勾選的同意方塊。正式上線前，主辦單位的個資或法遵人員必須核准完整文字、保存期限、利用對象與聯繫方式；未核准不得開放現場報名功能。

上述欄位依《個人資料保護法》第 8 條的告知事項設計。規格不取代主辦單位的法律審查。

## 12. 錯誤與邊界情況

- 同一手機或 E-mail 對應多筆資料：不顯示姓名，不自動報到，請洽工作人員。
- 手機與 E-mail 分別對應不同資料：不自動合併，請洽工作人員。
- 試算表缺少必要欄位或欄位順序不符：後端停止寫入並回傳 `SYSTEM_ERROR`。
- Bridge 載入失敗：顯示重新連線按鈕，不進入表單送出狀態。
- 使用者離線：保留當前表單記憶體狀態，連線恢復後由使用者再次送出。
- 確認期間他人已完成同一筆報到：回傳 `ALREADY_CHECKED_IN` 與原時間。
- 現場新增期間出現既有資料：不新增列，改回既有資料的遮罩確認流程。
- 時區錯誤：Apps Script 專案與試算表都必須設定為 `Asia/Taipei`；部署檢查不通過時禁止上線。

## 13. 測試與驗收

### 13.1 單元測試

- 手機、E-mail、姓名正規化與驗證。
- 各種長度姓名的遮罩結果。
- 快取分片與查詢命中、未命中、重建。
- 確認憑證建立、過期、重放與失效。
- 回應代碼與錯誤轉換。

### 13.2 整合測試

- 手機查到、手機查不到後 E-mail 查到、兩者都查不到。
- 預先報名確認、重複報到、現場報名及資料衝突。
- 兩個並行請求確認同一筆資料，只能寫入一次。
- 多個並行現場報名請求，不得建立重複資料。
- Bridge 拒絕非允許來源及格式錯誤訊息。
- 快取清除後仍可從 Google Sheet 得到正確結果。

### 13.3 介面測試

- 320、375、390、430、768 及 1440px 寬度。
- Android Chrome 與 iOS Safari 的數字鍵盤、E-mail 鍵盤及輸入焦點。
- 鍵盤操作、螢幕閱讀器狀態提示、降低動態效果及文字放大。
- 離線、慢速網路、Bridge 載入失敗與 60 秒虛擬等候室流程。

### 13.4 上線前壓力驗證

- 使用測試試算表及匿名假資料執行，不使用正式個資。
- 分別測試集中查詢與集中寫入，確認錯誤率、P95 回應時間、重複列數及鎖定失敗處理。
- 驗收標準：所有成功寫入皆唯一；忙碌請求能進入等候室；系統不洩漏完整個資；壓力測試不得產生重複報到或重複現場名單。
- Apps Script 配額會隨 Google 政策調整，壓力測試結果只代表測試當下，活動日前必須再執行一次冒煙與併發測試。

## 14. 部署與營運檢查

1. 在測試試算表完成欄位初始化與假資料測試。
2. 將 Apps Script 專案與試算表時區改為 `Asia/Taipei`。
3. 設定 Spreadsheet ID、工作表名稱、GitHub Pages 允許來源及確認憑證設定。
4. 以「執行身分：部署者」及「存取權：任何人」部署 Web App。
5. 將 Web App URL 寫入 GitHub Pages 前端設定。
6. 部署 GitHub Pages，確認 Bridge 握手、來源驗證及所有流程。
7. 由主辦單位核准 Logo、活動文字及個資蒐集告知。
8. 匯入正式名單，檢查重複手機、重複 E-mail、欄位格式及人數。
9. 清除並預熱查詢快取。
10. 使用實際 QR Code 進行 Android 與 iPhone 端到端測試。
11. 活動期間由工作人員監看 Apps Script 執行紀錄與試算表異常，不在公開畫面顯示技術錯誤內容。
